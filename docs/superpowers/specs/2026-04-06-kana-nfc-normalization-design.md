# 日本語仮名 NFC正規化 設計書

## Context

**Issue**: zrn-ns/crosspoint-reader#14
**問題**: 「ファイル一覧」画面でファイル名の濁音が分離して表示される（例: 「完全カ゛イト゛」→ 本来「完全ガイド」）

### 根本原因

SDカード上のファイル名がNFD（分解形）で格納されている場合、濁点(U+3099)や半濁点(U+309A)が独立したコードポイントとして存在する。`FileBrowserActivity`はSdFatから取得したファイル名をそのまま表示しており、NFC正規化を行っていないため、結合文字が独立した文字として描画される。

macOSはファイルシステムにNFD形式でファイル名を書き込むため、macOSからSDカードにコピーしたファイルで顕著に発生する。

### 「最近開いた本」画面で問題が起きない理由

`RecentBooksStore`はEPUBメタデータ（タイトル）を使用しており、ファイルシステムのファイル名を直接表示していない。

## 設計

### 新規関数

`lib/Utf8/Utf8.h`に以下のユーティリティ関数を追加する:

```cpp
/// 日本語仮名の濁点(U+3099)・半濁点(U+309A)をin-placeでNFC合成する。
/// 合成後は文字列が短くなる（結合文字3バイトが消えるため）。
void utf8NfcNormalizeKana(std::string& str);
```

### 合成対象

Unicode標準に基づき、以下のペアを合成する:

**濁点(U+3099)との合成** — ひらがな・カタカナの該当文字（約42ペア）:
- か→が, き→ぎ, く→ぐ, け→げ, こ→ご
- さ→ざ, し→じ, す→ず, せ→ぜ, そ→ぞ
- た→だ, ち→ぢ, つ→づ, て→で, と→ど
- は→ば, ひ→び, ふ→ぶ, へ→べ, ほ→ぼ
- う→ゔ（ひらがな）, ウ→ヴ（カタカナ）
- ワ→ヷ, ヰ→ヸ, ヱ→ヹ, ヲ→ヺ（カタカナのみ）
- 同パターンのカタカナ版すべて

**半濁点(U+309A)との合成** — は行のみ（10ペア）:
- は→ぱ, ひ→ぴ, ふ→ぷ, へ→ぺ, ほ→ぽ
- ハ→パ, ヒ→ピ, フ→プ, ヘ→ペ, ホ→ポ

### 実装方式

`static constexpr`なルックアップテーブル（baseコードポイント → 合成済みコードポイント）をFlashに配置。テーブルサイズは約50エントリ × 8バイト ≈ 400バイト。

処理アルゴリズム:
1. UTF-8文字列を先頭から走査
2. 現在のコードポイントがU+3099またはU+309Aの場合:
   a. 直前のコードポイントとの合成をテーブルで検索
   b. 合成可能なら、直前の文字を合成済みコードポイントに置換し、結合文字を削除
   c. 合成不可能なら、そのまま残す
3. in-place変換（結合文字削除で文字列が縮む）

### リソース影響

- **Flash**: +400バイト程度（ルックアップテーブル + 関数コード）
- **RAM**: 追加ヒープ割り当てなし（in-place変換）
- **CPU**: ファイル一覧ロード時のみ実行（ディレクトリ走査1回につき1回）

### スコープ外

- 全角/半角カナのNFKC正規化（U+FF65-FF9F）
- ラテン系アクセント記号のNFC正規化
- WebDAVやその他のファイル名表示箇所（今回はFileBrowserActivityのみ）
